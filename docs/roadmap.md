# Roadmap

这份文档只回答一个问题：

`这个 starter 跑通以后，下一步最推荐怎么扩展？`

它不是大而全产品规划。
它是给零基础用户和 AI 协作工具看的阶段路线图。

当前这个仓库，可以先理解成：

- 一个基于现有 OpenClaw 单 agent 状态，快速扩成多 agent 的 starter
- 当前先用 5-agent 内容架构做默认参考实现
- 后期再逐步走向更可定制、可交付、可按客户需求搭建的多 agent 方案

## 默认原则

- 先跑通，再扩展。
- 先本地，再外部。
- 先一个稳定流程，再拆更多场景。
- 先小改，再大改。
- 每一阶段都要能说明“现在已经能稳定做什么”。

## 阶段 0：先装起来

目标：把 starter 安装成功，并确认 5 个角色都在。

要做到：

- 能运行 `bash scripts/install.sh`
- 能运行 `bash scripts/check.sh`
- 知道 `boss`、`material`、`creator`、`thinktank`、`tech` 分别做什么

完成标志：

- 自检通过
- 能打开控制台
- 能向当前默认 boss agent（通常是 `<当前前缀>-boss`）发第一条消息

## 阶段 1：先跑通一条最小内容工作流

目标：不是做很多功能，而是完整走通一次“素材 -> 选题 -> 大纲 -> 初稿”。

推荐顺序：

1. 导入一篇公开素材
2. 更新素材索引
3. 让 thinktank 给可写角度
4. 让 creator 生成大纲
5. 让 creator 生成初稿

完成标志：

- 至少成功产出 1 份草稿
- 你知道默认 6 步流程在做什么
- 你知道哪个问题该找哪个 agent

## 阶段 2：再改成更像你自己的系统

目标：先改风格，不先改大结构。

最推荐先改：

- `templates/content-system/config/style-guide.md`
- `templates/workspaces/*/SOUL.md`
- `templates/workspaces/*/TOOLS.md`

这一阶段适合做：

- 改文风
- 改输出格式
- 改提示方式
- 补你常用的平台和素材类型

先不要急着做：

- 改角色名字
- 一口气加很多新 agent
- 复杂自动发布

## 阶段 3：让日常使用更稳定

目标：让这个系统不只是“能跑一次”，而是“能重复使用”。

推荐补强点：

- 固定一套你常用的首轮提示词
- 明确每个角色的边界
- 把常见报错整理成排障清单
- 把常见内容题材沉淀成模板

完成标志：

- 同样的流程你能重复跑多次
- AI 给你的下一步建议开始稳定
- 你知道哪些地方需要你人工确认，哪些可以放心交给 agent

## 阶段 4：先扩一个新场景，不要同时扩很多

目标：在现有内容工作流上，只新增 1 个实际场景。

推荐场景示例：

- 素材研究型：更强的资料搜集、索引、知识归档
- 选题策划型：更强的话题判断、标题方案、内容排期
- 发布准备型：把草稿整理成适合不同平台的版本
- 技术维护型：更强的配置检查、迁移、排障和升级支持

推荐做法：

- 先明确这个新场景到底解决什么问题
- 再决定是新增 skill、改现有角色，还是新加角色
- 每次只改一层，不要同时改流程、角色名、目录结构

## 阶段 5：再走向多场景 multi-agent 架构

目标：从“一个内容 starter”扩到“多个应用场景共存的 agent 系统”。

这时再考虑：

- 不同场景是否需要不同 boss
- 是否要按场景拆新的 workspace
- 是否要给不同 agent 配不同模型
- 是否要接不同渠道或不同账号
- 是否要做更严格的权限和 sandbox 隔离

这一步更像“系统设计”，不再只是“提示词改造”。

推荐顺序：

1. 先确定一个主入口
2. 再确定每个场景的边界
3. 再决定哪些能力复用，哪些独立
4. 最后才做复杂路由和自动化

这一步也更接近你未来的交付方向：

- 自己快速搭建
- 远程协助客户搭建
- 让客户自己跑简单安装命令
- 后期再考虑更像“软件”的交付形式

## 现在不推荐太早做的事

- 一开始就做几十个 agent
- 一开始就给每个 agent 配不同模型
- 一开始就上外部自动发布
- 一开始就照搬别人的完整架构
- 一开始就同时改目录、角色名、工作流和配置

原因很简单：

- 新手最容易在复杂度里迷路
- 出问题时很难判断是哪一层坏了
- 你会更难学会“为什么这样设计”

## 如果你想参考成熟项目

可以。

但默认策略是：

- 先学习它为什么这样拆角色
- 再判断哪些部分适合你现在这一阶段
- 最后只吸收最有用的一小部分

不要把“看到一个成熟项目”直接等同于“现在就要做成那样”。

## 当前最推荐的一条路

如果你现在还是零基础，最推荐顺序是：

1. 先装好 starter
2. 先跑通一篇内容
3. 先改成更像你的文风和流程
4. 再只扩 1 个新场景
5. 最后再做多场景 multi-agent 架构

一句话总结：

`先让一个小系统稳定可用，再把它慢慢长成更大的系统。`
