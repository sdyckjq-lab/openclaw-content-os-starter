# Local Reference Rules

这份文档只回答一个问题：

`当我拿一个成熟项目来参考时，AI 应该怎么用它？`

目标不是复制外部项目。
目标是先学习、再比较、再决定是否局部复用。

## 默认原则

- 参考项目默认只用于本地学习和设计参考。
- 先分析能不能借鉴，再决定要不要实现。
- 优先复用思路，不优先复制代码。
- 优先最小改造，不优先整套搬运。
- 除非用户明确要求，不要把外部项目内容直接并入、复制或上传到当前仓库。

## 什么算“参考项目”

包括但不限于：

- 你本地克隆的开源仓库
- 你下载下来的 demo
- 你之前做过的旧项目
- 你想学习的成熟 agent 系统

## AI 使用参考项目的标准流程

### 第一步：先看它是干什么的

先说明：

- 这个项目的目标是什么
- 它解决的是哪一类问题
- 它和当前 starter 的相同点是什么
- 它和当前 starter 的不同点是什么

### 第二步：再判断值不值得学

至少回答这几个问题：

- 它的结构是否比当前仓库更成熟
- 它适不适合零基础阶段直接采用
- 它是“能局部借鉴”，还是“只适合后期参考”
- 它会不会把当前 starter 复杂度一下拉太高

### 第三步：明确建议怎么用

输出时要尽量说明：

- 哪些值得学
- 哪些不适合直接搬
- 哪些可以改一改后再用
- 哪些最好先别碰

### 第四步：如果真的要复用，只做最小复用

优先顺序：

1. 先复用思路
2. 再复用结构
3. 最后才考虑复用具体实现

如果只是为了学习，通常只需要前两层。

## AI 输出时必须说清楚的 4 件事

1. 参考项目里最值得学的点是什么
2. 当前仓库最适合吸收的是哪一小部分
3. 如果照搬，会带来什么额外复杂度
4. 现在最推荐的下一步是什么

## 不要这样用参考项目

- 不要看到成熟项目就默认整套照搬
- 不要把不属于当前仓库的工程化工具链硬塞进来
- 不要把外部项目里的密钥、账号、私有路径、私有数据带进当前仓库
- 不要把外部项目的目录结构直接覆盖当前 starter
- 不要把“学习参考”变成“偷偷复制到当前 GitHub 仓库”

## 可以这样用参考项目

- 学角色拆分方式
- 学文档结构
- 学提示词组织方式
- 学 agent 之间怎么交接
- 学哪些能力值得拆成独立 skill
- 学不同阶段的扩展顺序

## 以 `agency-agents` 为例

如果你需要更细的学习顺序，继续看：

- `docs/agency-agents-learning-checklist.md`

它适合学习的地方：

- 角色拆分很细
- 每个 agent 的职责写得比较完整
- 有多 agent 协作示例
- 很适合后期研究“复杂编排长什么样”

它不适合直接做的事：

- 不适合现在就整套搬进这个 starter
- 不适合零基础阶段直接照着做成几十个 agent
- 不适合把它当成当前仓库的直接替代品

更合适的用法是：

- 先学它怎么写 agent 角色
- 先学它怎么组织多 agent 协作
- 再只把最适合当前阶段的一小部分吸收到本项目

## 如果用户明确要求“参考后直接改造”

也不要直接动手复制。

应该先做这三步：

1. 说明准备借鉴哪一部分
2. 说明为什么这部分适合当前仓库
3. 说明改造后会影响哪些文件和结构

然后再做最小必要修改。

## 和公开仓库安全规则的关系

这份文档不改变公开仓库安全约束。

始终保持：

- 不提交真实密钥
- 不提交私有素材
- 不提交私有账号信息
- 不把本地参考项目内容未经确认就放进当前公开仓库

## 一句话规则

`参考项目先用来学，再用来比，最后才决定要不要只改一小部分。`
