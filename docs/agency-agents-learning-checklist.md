# Agency-Agents Learning Checklist

这份清单只回答一个问题：

`如果把 agency-agents 当本地学习参考，我现在最该学什么？`

目标不是照搬它。
目标是把它拆成你当前阶段能吸收的部分。

## 先记住它是什么

`agency-agents` 更像一个大型 agent 角色库。

它的特点是：

- 角色很多
- 分工很细
- 每个 agent 写得比较完整
- 有多 agent 协作示例
- 已经开始接近复杂编排系统

对你现在最有价值的，不是“数量多”。
而是“它怎么把一个 agent 写清楚”。

## 你当前项目最需要学的 4 件事

### 1. 学它怎么定义一个 agent

先看这些点：

- 这个 agent 是干什么的
- 它负责什么，不负责什么
- 它输出什么结果
- 它成功时应该长什么样

你当前仓库最适合吸收的位置：

- `templates/workspaces/*/AGENTS.md`
- `templates/workspaces/*/SOUL.md`
- `templates/workspaces/*/TOOLS.md`

当前最推荐做法：

- 不是新增很多 agent
- 而是先把现有 5 个角色写得更稳定、更清楚

### 2. 学它怎么组织多 agent 协作

重点不是“并行很酷”。
重点是：

- 谁是入口
- 谁负责任务拆分
- 谁产出中间结果
- 谁做最终汇总

这和你当前仓库最相关的地方是：

- `docs/architecture.md`
- `templates/workspaces/boss/AGENTS.md`

你现在最该学的是：

- 如何把 `boss` 写成稳定入口
- 如何把 5 个角色的边界说清楚
- 如何减少“本来该分派，结果 boss 全自己做完”的情况

### 3. 学它怎么写 use case（使用场景）

`agency-agents` 的价值之一，是它不只写 agent，还会写“这些 agent 一起怎么用”。

这对你后面做客户定制很有帮助。

因为你未来真正卖的，不只是 5 个角色。
而是：

- 一个客户场景该怎么拆
- 哪些角色要保留
- 哪些角色要改
- 哪些技能要接进去

你当前可以先学会：

- 每个场景只保留一个清晰目标
- 每个场景只先做一条最小流程
- 每个场景都能解释“为什么这样拆角色”

### 4. 学它怎么做“后期扩展”，不是“起步安装”

它更适合当后期参考。

因为你当前项目真正要守住的是：

- 用户已经有单 agent OpenClaw
- 想快速装成多 agent
- 要尽量简单
- 要尽量少报错
- 要能按客户需求继续定制

所以你更应该学它的：

- 角色设计
- 协作设计
- 场景设计

而不是直接学它的：

- 超大规模 agent 数量
- 很重的编排流程
- 偏复杂的完整研发流水线思路

## 现在最值得学的部分

如果只学一小部分，优先顺序是：

1. agent 职责写法
2. agent 边界写法
3. 多 agent 协作示例
4. 场景化案例组织方式

这 4 个最容易直接转化成你自己的东西。

## 现在先不要学太深的部分

先不要把重点放在这些地方：

- 一次并行调很多 agent
- 把所有场景都做成完整 orchestrator
- 一开始就做几十个角色分类
- 一开始就用它的复杂质量门和完整流水线思路改当前仓库

原因：

- 你现在最重要的是“快速搭起来”
- 不是“先变成最复杂的系统”

## 和你当前项目目标怎么对应

你现在这个项目的长期方向可以理解成：

- 先用当前 5-agent 架构做一个默认参考实现
- 再让已有 OpenClaw 单 agent 状态的用户，能更快转成多 agent 架构
- 再根据不同客户需求，做定制化搭建

所以从 `agency-agents` 最值得借鉴的是：

- 如何把角色写得更容易改
- 如何让一个场景更容易替换角色组合
- 如何让案例更容易复用

而不是把它整个搬过来。

## 你可以按这个顺序学

### 第一轮：只学结构

只回答这些问题：

- 它一个 agent 文档一般包含什么
- 它怎么写职责和边界
- 它怎么写 workflow

输出应该是：

- 你自己的 5 个角色哪里写得还不够清楚

### 第二轮：只学协作

只回答这些问题：

- 它怎么描述多个 agent 一起完成一件事
- 它怎么让不同角色输出能接起来

输出应该是：

- 你自己的 `boss -> 其他角色 -> boss` 哪些地方需要补规则

### 第三轮：只学场景化

只回答这些问题：

- 它怎么把 agent 组合成一个使用场景
- 它怎么解释“这个场景为什么这样搭”

输出应该是：

- 你以后给不同客户做定制时，哪些场景可以沉淀成模板

### 第四轮：最后才学复杂编排

这一轮再看：

- orchestrator 思路
- 更复杂的质量门
- 更长的 agent 流水线

这时你会更容易判断：

- 哪些值得吸收
- 哪些会把你当前 starter 变得太重

## AI 参考时应该给你的输出格式

以后如果 AI 去看 `agency-agents`，更推荐这样回给你：

1. 这次最值得学的 3 个点
2. 哪 1 个点最适合当前仓库先吸收
3. 如果现在照搬，会增加什么复杂度
4. 最推荐先改哪一个文件

## 一句话总结

`把 agency-agents 当“后期结构老师”，不要把它当“现在就整套照搬的模板”。`
