# Reference Projects

这份文档只回答一个问题：

`以后如果要继续扩这个项目，优先从哪些成熟项目借鉴，哪些东西甚至可以直接拿来用？`

这份文档写给未来的你，也写给未来接手的 AI。

## 总原则

这个仓库未来明确不追求“什么都自己从零写”。

默认策略是：

- 先看外部成熟项目有没有现成可借的内容
- 优先借结构、模板、角色写法、安装思路
- 能局部直接复用时，就不要重复造轮子
- 只有在当前仓库目标明显不匹配时，才自己重新写

前提是：

- 保持公开仓库安全
- 保持当前 starter 的新手友好定位
- 不把复杂度一下拉太高

## 当前优先参考的两个项目

### 1. `awesome-openclaw-agents`

链接：

- <https://github.com/mergisi/awesome-openclaw-agents>

定位判断：

- 更像 OpenClaw 的 `SOUL.md` 模板大全
- 强项是单 agent 模板数量多、覆盖场景广
- 适合快速找“某个角色的人格模板”或行业化角色灵感

最值得借的地方：

- 现成 `SOUL.md` 模板
- 场景分类方式
- 角色命名方式
- “用户一看就懂” 的角色描述方式

什么情况下可以直接拿来用：

- 你需要一个新场景里的单个角色人格
- 你当前仓库还没有这个角色的好版本
- 它的语气、目标、边界和你场景比较贴近

最推荐的复用方式：

1. 先挑最接近的角色
2. 先局部复用 SOUL 结构或段落
3. 再按当前 starter 的角色边界改写

不推荐直接照搬的地方：

- 它不是你的场景化多 agent 安装骨架
- 它不是你当前仓库的 installer / check / sandbox 方案
- 它不是“用户选场景后一键部署”的完整 starter

一句话定位：

`它是角色模板库，不是你的整体产品骨架。`

### 2. `agency-agents`

链接：

- <https://github.com/msitarzewski/agency-agents/tree/main>

定位判断：

- 更像大型 agent 角色库 + 多工具适配器
- 强项是角色文档成熟、边界清晰、协作说明更完整
- 还有把 agent 转成 OpenClaw / 其他工具格式的思路

最值得借的地方：

- agent 文档结构
- 职责和边界的写法
- 多角色协作描述方式
- 多工具导出 / 转换思路
- OpenClaw 适配思路

什么情况下可以直接拿来用：

- 你要补某个角色的 `AGENTS.md` / `SOUL.md` 写法
- 你要研究未来怎么把角色库导出成不同工具格式
- 你要参考它的 OpenClaw 集成思路，而不是从零想一遍

最推荐的复用方式：

1. 先借“怎么写清一个 agent”
2. 再借“怎么描述多 agent 协作”
3. 最后才考虑借转换脚本思路

不推荐现在直接照搬的地方：

- 超大角色数量
- 偏重的复杂协作体系
- 直接把它当成当前仓库的完整替代方案

一句话定位：

`它是结构和写法老师，不是现在就该整套搬进来的成品。`

## 未来任务中的默认复用顺序

以后如果用户说：

- “我要加一个新场景”
- “我要加一个新角色”
- “我要让角色更专业”
- “我要让这个仓库更像成熟项目”

默认先按这个顺序判断：

1. 当前仓库里有没有现成角色 / skill / 模板可复用
2. `awesome-openclaw-agents` 里有没有现成 SOUL 模板可借
3. `agency-agents` 里有没有更成熟的角色说明或协作写法可借
4. 如果都没有，再自己从零实现

## 可以直接抄什么，不建议抄什么

### 可以优先考虑直接复用的

- 角色人格模板
- 角色职责写法
- 场景分类方式
- 文档组织结构
- 局部 OpenClaw 集成思路

### 不建议直接整套照搬的

- 整个目录结构
- 整个安装器逻辑
- 整套复杂 agent 体系
- 不符合当前 starter 轻量定位的工程化流程

## AI 工具以后接手时应默认怎么说

如果未来 AI 去看这两个项目，默认输出应该包含：

1. 哪个项目更适合这次任务
2. 最值得复用的是哪一小部分
3. 如果直接照搬会多出什么复杂度
4. 当前最推荐的最小复用方案是什么

## 当前最推荐的借鉴路线

如果只借一小部分，优先顺序建议是：

1. 先从 `awesome-openclaw-agents` 借角色人格模板
2. 再从 `agency-agents` 借角色边界和协作写法
3. 再考虑未来是否借它的转换 / 多工具适配思路

这和你当前项目方向最一致：

- 不是卷角色数量
- 而是做“场景化、一键安装、可交付”的 starter

## 一句话总结

`未来扩展时，优先借现成角色和成熟写法，不要把“自己从零重写一遍”当默认选项。`
