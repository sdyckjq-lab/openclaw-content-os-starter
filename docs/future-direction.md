# Future Direction

这份文档只回答一个问题：

`这个项目未来到底想长成什么样，AI 以后应该按什么方向继续推进？`

## 当前定位

当前这个仓库先不要理解成“一个内容创作小工具”。

更准确地说，它现在是：

- 一个面向新手的 OpenClaw 多 agent 场景化 starter
- 当前先用“内容创作”做第一个默认参考实现
- 后续再逐步扩成多个场景可选的一键安装仓库

## 长期目标

长期目标不是：

- 做成最多 agent 的模板仓库
- 和大型 agent 角色库拼数量

长期目标是：

- 用户选择一个场景
- 跑一条安装命令
- 直接得到一套可用的多 agent 系统
- 后续还能继续按客户需求定制

也就是说，你未来真正要提供的东西是：

`场景化、一键安装、可维护、可交付的 OpenClaw 多 agent 方案。`

## 未来要坚持的 6 个原则

### 1. 先装起来，再谈高级能力

任何新增场景，都先保证：

- 安装清楚
- 自检清楚
- 最小流程清楚

### 2. 先复用，再新增

以后默认优先顺序：

1. 复用当前仓库已有角色和 skill
2. 参考外部成熟项目做局部借鉴
3. 最后才自己从零写

### 3. 先一个场景，再多个场景

不要一口气扩很多新场景。

更推荐：

- 一次只做 1 个新 preset
- 先把这个 preset 的最小路径跑通
- 再考虑下一个

### 4. 先本地路径，再外部渠道

新手默认快乐路径仍然应该是：

- 安装
- 自检
- dashboard
- 本地流程跑通

Telegram、发布、自动化都应该是第二层。

### 5. 先借成熟项目，再避免重复造轮子

未来任务中，如果外部已有成熟模板、角色、OpenClaw 适配思路：

- 优先借
- 优先改
- 优先拼装

不要默认“从零再写一份”。

### 6. 先可交付，再酷炫

这个项目未来更接近：

- 你自己快速搭建
- 远程协助客户搭建
- 给客户一条命令先装起来

所以“可理解、可维护、少报错”比“很炫的复杂编排”更重要。

## 当前最适合的近期路线

### 阶段 A：把当前 content-basic 打磨得更稳

重点：

- 安装结束提示更像用户视角
- boss 的入口规则更稳
- 5 个角色边界更清楚
- 默认工作流更少跑偏

### 阶段 B：把“怎么扩一个新场景”模式化

重点：

- 明确 preset 结构
- 明确哪些层可复用
- 明确新增 preset 的最小检查单

### 阶段 C：新增第 2 个真实场景

推荐选：

- 和内容系统邻近
- 能复用现有骨架
- 但边界又足够清楚

比如：

- 素材研究型
- 选题策划型
- 发布准备型

### 阶段 D：逐步做成“多场景 starter 仓库”

到这一步时，再考虑：

- preset 列表页
- preset 选择提示
- 角色库沉淀
- 更通用的转换 / 复用策略

## 以后新增实例时，AI 默认应怎么思考

以后如果 AI 被要求“加一个新实例 / 新场景”，默认应该先回答：

1. 这个场景的用户是谁
2. 当前仓库里哪些角色能复用
3. 哪些 skill 能复用
4. 外部成熟项目里有没有现成模板可借
5. 最小可安装版本是什么

只有这 5 个问题大致清楚，才值得开始改代码或改模板。

## 对参考项目的明确态度

对下面这两个项目，默认不是“只能学习不能用”。

而是：

- 可以学
- 可以比
- 可以局部直接拿来用
- 但不要盲目整套照搬

参考项目：

- `awesome-openclaw-agents`
- `agency-agents`

具体看：

- `docs/reference-projects.md`

## AI 以后做规划时的默认输出方式

如果以后 AI 帮你做下一步规划，最理想的输出不是一大堆平行选项。

而应该尽量包含：

1. 当前项目现在已经稳定能做什么
2. 这次最推荐只推进哪一个小目标
3. 为什么先做它
4. 哪些东西可以直接复用
5. 最小验证路径是什么

## 一句话总结

`未来方向不是卷 agent 数量，而是把这个仓库做成“场景可选、装完能用、尽量复用成熟成果”的 OpenClaw 多 agent starter。`
